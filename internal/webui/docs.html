Overview

karots-drop is an ephemeral text and file sharing tool. Data is stored in-memory with a configurable TTL and a 6-digit code. No database, no persistence, no accounts. Share from the CLI or the Web UI — recipients retrieve data with a simple code.

CLI Reference

serve

Start the HTTP server.

karots-drop serve [flags]
FlagDefaultDescription
--addr:8080Listen address and port
--bind""Bind to interface (localhost restricts to 127.0.0.1)
--token""API token for X-Auth-Token header
--rate-limit60Max requests per minute per IP (0 = unlimited)
--max-items500Max items in store (0 = unlimited)
--delete-on-retrievefalseDelete items immediately after retrieval
--ttl20mItem TTL (e.g. 5m, 30m, 1h)

send

Upload text, a file, or piped data.

karots-drop send [flags] "text"
echo "hello" | karots-drop send
karots-drop send --file photo.jpg
karots-drop send --encrypt "secret"
karots-drop send --ttl 5m "short-lived"
karots-drop send --compact "data"
FlagDefaultDescription
--file""Path to a file to upload
--serverhttp://localhost:8080Server URL
--encryptfalseEncrypt with AES-256-GCM before upload
--qrfalseDisplay QR code in terminal
--jsonfalseJSON output
--ttl""Request custom TTL (capped at server max)
--compactfalseOutput only the code (scripting)

get

Retrieve data by its 6-digit code.

karots-drop get [flags] <code>
karots-drop get 482910 --json
karots-drop get 482910 --key "a1b2c3...base64..."

clip

Upload clipboard content (Linux, requires xclip or xsel).

karots-drop clip
karots-drop clip --encrypt
karots-drop clip --watch

version & health

karots-drop version
karots-drop health

completion

source <(karots-drop completion)

REST API

POST /api/store

Upload text or a file. Accepts multipart/form-data or raw body.

curl -X POST -d "hello world" http://localhost:8080/api/store
curl -X POST -F "text=hello" http://localhost:8080/api/store
curl -X POST -F "file=@photo.jpg" http://localhost:8080/api/store
curl -X POST -F "text=hello" -F "ttl=5m" http://localhost:8080/api/store

Response 201 Created:

{
  "code": "482910",
  "ttl": 1200,
  "url": "http://localhost:8080/api/get/482910",
  "filename": "photo.jpg",
  "encrypted": false
}

GET /api/get/{code}

Retrieve stored data by code. Returns raw bytes with original Content-Type. When --delete-on-retrieve is enabled, the item is deleted after reading.

curl http://localhost:8080/api/get/482910

GET /api/qr/{code}

Returns a 256x256 PNG QR code encoding the retrieval URL.

curl -o qr.png http://localhost:8080/api/qr/482910

GET /api/health

Health check. Returns {"status":"ok"}. Exempt from auth.

curl http://localhost:8080/api/health

Security Model

  • Data at rest: In-memory only, no persistence to disk
  • Data in transit: No built-in TLS — deploy behind Caddy or nginx for HTTPS
  • Encryption: Optional AES-256-GCM, client-side key (zero-knowledge)
  • Access control: Optional --token flag. Web UI reads token from ?token= query param
  • Upload limit: 20 MB per item
  • TTL: Configurable (default 20 min), enforced server-side
  • Rate limiting: Per-IP sliding window (default 60 req/min)
  • Store capacity: Configurable max items (default 500)
  • Restart: All data lost on restart (by design)

Deployment

Docker

docker build -t karots-drop .
docker run -d -p 8080:8080 karots-drop serve \
  --addr :8080 --token "$TOKEN" --rate-limit 30 \
  --max-items 200 --ttl 10m --delete-on-retrieve

systemd

[Unit]
Description=karots-drop
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/karots-drop serve --addr :8080 --ttl 30m
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target

Build from source

make build
make build VERSION=1.0.0
make test
make clean

FAQ

Why not use a database?

By design: no persistence across restarts means no files to manage, no backup concerns, no stale data cleanup. The TTL handles expiry automatically.

How are codes generated?

crypto/rand picks 6 decimal digits. If there's a collision (extremely rare), the server retries up to 100 times.

How does delete-on-retrieve work?

The item is removed immediately after reading. A second GET for the same code returns 404.

Can I use custom TTL per upload?

Yes. Clients can request a shorter TTL with karots-drop send --ttl 5m or the ttl form field. The server caps it at its own maximum.

How do I get just the code for scripting?

Use karots-drop send --compact "data" — it prints only the 6-digit code.

Is there TLS support?

Not built-in. Deploy behind Caddy, nginx, or a Cloudflare Tunnel for HTTPS.
